# PascalCompiler

`PascalCompiler` это учебный компилятор под подмножество Pascal на Java. Я собрал его по этапам курса: сначала сделал разбор программы и AST, потом добавил семантику и JVM backend, а на третьем этапе реализовал оптимизации и собственную стековую виртуальную машину. Сейчас проект проходит весь путь от `.pas` файла до исполняемой программы двумя способами: через JVM и через свою VM.

## Поддерживаемое подмножество Pascal

Компилятор работает со следующим набором конструкций:

- типы `integer`, `double`, `boolean`, `char`, `string`
- одномерные массивы `array[..] of ...`
- объявления `var`, `const`, `procedure`, `function`
- операторы `begin/end`, `if/then/else`, `for`, `while`, `repeat/until`
- управляющие конструкции `break`, `continue`, `exit`
- ввод и вывод через `write`, `writeln`, `read`, `readln`
- встроенные функции `inc`, `dec`, `abs`, `length`, `pos`, `copy`

Я не добавлял `record`, пользовательские `type`, многомерные массивы, вложенные процедуры и `var` параметры в backend'ах. Для этой работы было важнее довести выбранное подмножество до законченного состояния, чем распыляться на лишние возможности.

## Что сделано в 1 аттестации

На первом этапе я собрал базу компилятора.

Сначала появился лексический анализатор. Он читает исходный Pascal код и разбивает его на токены: ключевые слова, идентификаторы, литералы, операторы и разделители.

После этого я реализовал синтаксический анализатор. Он строит AST, то есть абстрактное синтаксическое дерево программы. В этом дереве уже видна структура программы: объявления, выражения, циклы, условия, вызовы функций и присваивания.

Отдельно я добавил диагностику синтаксических ошибок. Если во входной программе пропущен символ, неверно записан оператор или сломана структура блока, компилятор сообщает позицию ошибки и ожидаемую конструкцию.

## Что сделано во 2 аттестации

На втором этапе компилятор начал понимать смысл программы, а не только её форму.

Я добавил семантический анализ. Он проверяет, что все идентификаторы объявлены, типы используются правильно, функции вызываются с корректным числом аргументов, условия в `if` и `while` имеют булев тип, а присваивания не нарушают правила типов.

После семантического анализа AST становится типизированным. Узлы дерева получают `resolvedType` и, где нужно, `resolvedSymbol`. Это важно для следующих стадий, потому что backend уже работает по дереву, в котором известны типы выражений, переменных и функций.

В конце второго этапа я добавил backend под JVM. Компилятор генерирует Java код, затем вызывает `javac`, компилирует результат и запускает программу через `java`. Это дало полный рабочий путь от Pascal файла до реально выполняемого кода.

## Что сделано в 3 аттестации

На третьем этапе я довёл проект до законченного вида и добавил собственную виртуальную машину.

Сначала я вынес общую логику компиляции в отдельный pipeline. Теперь `Main` не держит всю работу внутри себя, а только читает аргументы, запускает нужные этапы и выбирает backend.

Потом я добавил этап оптимизации AST. Оптимизатор упрощает локальные выражения, убирает константные ветки, удаляет мёртвые циклы и вырезает недостижимый код после `exit`, `break` и `continue`.

Главное расширение третьей аттестации это backend под простую стековую VM. Компилятор теперь умеет переводить программу в текстовый `.pvm` код, загружать этот код обратно и исполнять его внутри своей виртуальной машины.

Я также дополнил проект тестами и собрал документацию в одном месте, чтобы на защите можно было показать не только код, но и внятное описание архитектуры.

## Общая логика работы

Проект проходит такие стадии:

```text
Pascal source
 -> Lexer
 -> Parser
 -> AST
 -> SemanticAnalyzer
 -> AstOptimizer
 -> backend
```

Полная цепочка выглядит так:

1. Пользователь передаёт `.pas` файл.
2. `Lexer` разбивает текст на токены.
3. `Parser` строит AST.
4. `SemanticAnalyzer` проверяет семантику и типизирует дерево.
5. `AstOptimizer` упрощает AST, если включён уровень оптимизации.
6. Компилятор выбирает backend.
7. Программа либо запускается, либо сохраняется как промежуточный артефакт.

## JVM backend

JVM backend остался в проекте как результат второй аттестации.

Он работает так:

```text
Pascal -> typed AST -> Java source -> javac -> java
```

Компилятор генерирует Java файл, кладёт его в `build/generated`, потом вызывает `javac`, а затем запускает скомпилированный класс через `java`. Этот путь удобно показывать на защите, потому что он даёт понятный и наглядный исполняемый результат.

## VM backend

VM backend это основная часть третьей аттестации.

Он работает так:

```text
Pascal -> typed AST -> .pvm listing -> loader -> VmRuntime
```

После семантики и оптимизации компилятор передаёт AST в VM compiler. Тот переводит дерево в стековые инструкции и записывает их в текстовый `.pvm` файл. Потом loader читает этот файл, собирает внутреннюю модель программы, а `VmRuntime` исполняет её.

Виртуальная машина использует простую стековую модель:

- `operand stack` для вычисления выражений
- `call stack` для вызова функций и процедур
- глобальные слоты для глобальных переменных
- локальные слоты для параметров, локальных переменных и временных значений

Функция возвращает результат через выделенный локальный слот с именем функции. Инструкция `RET` читает это значение и передаёт его вызывающему коду.

## Формат `.pvm`

`.pvm` это текстовый listing, который удобно читать глазами. В нём есть:

- заголовок программы
- точка входа
- глобальные слоты
- routines
- инструкции

Пример:

```text
program Demo
entry __main

globals 1
global 0 total integer

routine __main void 0 -1
instr PUSH_CONST integer 2
instr STORE_GLOBAL 0
instr LOAD_GLOBAL 0
instr PRINTLN
instr HALT
end
```

Этот формат полезен и как промежуточное представление, и как наглядная демонстрация того, что именно сгенерировал backend.

## Какие оптимизации есть

Оптимизатор работает на уровне AST до генерации backend'ов. Поэтому и JVM backend, и VM backend получают уже упрощённую программу.

Сейчас реализованы:

- `constant folding` для арифметики, сравнений, булевой логики и строковых литералов
- упрощения `x + 0`, `x - 0`, `x * 1`, `x * 0`, `x / 1`
- булевы упрощения `not not x`, `true and x`, `false or x`, `false xor x`
- удаление веток `if true` и `if false`
- удаление цикла `while false`
- удаление недостижимого хвоста после `exit`, `break`, `continue`

Для VM это тоже считается оптимизацией. Я не добавлял отдельный peephole оптимизатор по байткоду, но VM backend работает уже по оптимизированному AST, поэтому в `.pvm` не попадает часть лишних инструкций.

## Как вывести AST

AST печатается по умолчанию. Чтобы его увидеть, достаточно запустить программу без флага `--no-ast`.

Пример:

```powershell
java -cp build/classes pascal.Main examples/optimizer_demo.pas
```

Если нужен только запуск без дерева, тогда используется `--no-ast`.

## CLI флаги

Основные флаги задаются в `pascal.Main`.

- `--tokens` — печать токенов
- `--no-ast` — отключить печать AST
- `--semantic` — флаг распознаётся, но отдельного поведения сейчас не добавляет
- `--run` — запуск JVM backend
- `--emit-java <dir>` — сохранить Java код
- `--run-vm` — запуск через собственную VM
- `--emit-vm <file>` — сохранить `.pvm`
- `--opt-level <0|1>` — выбрать уровень оптимизации

## Команды для запуска

Сборка компилятора:

```powershell
$sources = Get-ChildItem -Path "src/main/java" -Filter "*.java" -Recurse | ForEach-Object { $_.FullName }
javac -encoding UTF-8 -d build/classes $sources
```

Показать AST:

```powershell
java -cp build/classes pascal.Main examples/optimizer_demo.pas
```

Запуск через JVM:

```powershell
java -cp build/classes pascal.Main --no-ast --run tests/positive/arrays_math_control.pas
```

Запуск через VM:

```powershell
java -cp build/classes pascal.Main --no-ast --run-vm examples/vm_demo.pas
```

Сгенерировать `.pvm`:

```powershell
java -cp build/classes pascal.Main --no-ast --emit-vm build/generated/vm_demo.pvm examples/vm_demo.pas
```

Сравнить AST до и после оптимизации:

```powershell
java -cp build/classes pascal.Main --opt-level 0 examples/optimizer_demo.pas
java -cp build/classes pascal.Main examples/optimizer_demo.pas
```

Прогнать тесты:

```powershell
powershell.exe -ExecutionPolicy Bypass -File scripts/run_tests.ps1
```

## Тесты

В проекте есть:

- позитивные тесты на язык и оба backend'а
- негативные тесты на семантические ошибки
- тесты на VM
- тесты на оптимизатор
- ручный сценарий на ввод и вывод

Все основные проверки запускает `scripts/run_tests.ps1`. Скрипт сначала компилирует проект, потом по очереди прогоняет нужные сценарии.

## Ограничения

У проекта есть понятные ограничения:

- backend'ы не поддерживают `var` параметры
- вложенные процедуры и функции не реализованы
- `record`, пользовательские `type` и многомерные массивы отсутствуют
- VM использует текстовый `.pvm` формат, без бинарной сериализации
